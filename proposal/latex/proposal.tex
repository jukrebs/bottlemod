% 1.1 Problem statement
\subsection{Problem Statement}
Data-intensive scientific workflows spend a significant fraction of their execution time on I/O operations whose effective throughput depends on the operating system's page cache.
When the working set exceeds available memory, cache evictions force data to be re-read from slow block devices, creating performance cliffs that shift bottlenecks from compute to storage.
BottleMod~\cite{bottlemod} provides fast analytical bottleneck analysis but treats data availability as a single, constant rate, therefore it cannot represent the performance difference between cached and uncached data access.

% 1.2 Motivation
\subsection{Motivation}
Accurate bottleneck detection and attribution is important for informed resource allocation: a scheduler (machine or person) that is unaware of a memory-bound task will waste budget due to longer execution times instead of allocating more memory or reordering tasks.

\subsection{Related Work}

\textbf{WRENCH/SimGrid}: SimGrid is a framework for simulating executions of distributed applications on distributed platforms, and WRENCH builds on top of SimGrid by providing higher-level abstractions for workflow applications, services, and execution controllers. These systems are well suited for evaluating scheduling and runtime behavior on explicitly modeled platforms, but they primarily target discrete-event simulation of distributed scenarios rather than lightweight analytic bottleneck prediction. In contrast, BottleMod-CA keeps BottleMod's requirement/input formalism and augments it with storage tiers and page-cache-aware hit-rate models, enabling direct reasoning about task-level bottlenecks and cache-sensitive ordering effects with low modeling overhead. Hence, WRENCH/SimGrid are best viewed as complementary baselines: broader and more expressive for system-level workflow simulation, while BottleMod-CA targets interpretable per-task prediction of storage-hierarchy effects validated against real executions. \\

\subsection{Research Questions}
\begin{enumerate}[nosep,label=\textbf{RQ\arabic*:}]
	\item \textbf{Bottleneck identification.} Does the cache-aware extension to BottleMod correctly identify the limiting resource at each phase of task execution?
	\item \textbf{Prediction accuracy.} Can cache-aware BottleMod predict runtime of memory bound tasks more accurately than the original BottleMod paper.
\end{enumerate}

% 2.3 Scope
\subsection{Scope}

\subsubsection{In Scope}
\begin{itemize}[nosep]
	\item Single-node storage hierarchies: Linux page cache (DRAM) + block device (SATA HDD/SSD).
	\item Sequential and random I/O access patterns with varying working set sizes.
	\item Pluggable cache behavior models: direct hit-rate, stack-distance-based.
\end{itemize}

\subsubsection{Out of Scope}
\begin{itemize}[nosep]
	\item Distributed/networked storage.
	\item CPU cache hierarchy effects (L1/L2/L3)
	\item Kernel-level page cache modifications or custom eviction policies.
	\item Effects of IOPS on workflow performance.
\end{itemize}

\subsection{Solution Approach}
The extension introduces three process-side and environment-side constructs that map onto BottleMod's existing requirement/input function paradigm:

\begin{itemize}[nosep]
	\item \textbf{Logical Access Profile} $A_k^r(p), A_k^w(p)$: cumulative bytes read/written by dataset $k$ as a function of progress. Process-side descriptor provided by the developer.
	\item \textbf{Storage Tier} with bandwidth input $I_{bw,j}(t)$ and capacity $C_j$: environment-side descriptor of each tier $j$ (e.g., Memory at 1000\,MB/s, HDD at 100\,MB/s).
	\item \textbf{Tier Mapping} $H_{k,j}^r(p)$: fraction of dataset $k$'s accesses served by tier $j$ at progress $p$, computed by a pluggable \texttt{CacheBehaviorModel} (direct hit-rate, stack-distance CDF).
\end{itemize}

The key derivation is: $R'_{(k,j,bw,r)}(p) = H_{k,j}^r(p) \cdot {A_k^r}'(p)$, which converts hit fractions and access rates into per-tier bandwidth requirement functions.
These are appended to the standard \texttt{Task.cpu\_funcs} list, and the tier bandwidth inputs become \texttt{in\_cpu\_funcs} for \texttt{TaskExecution}.
